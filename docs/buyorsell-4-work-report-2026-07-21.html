2. Kiến Trúc 5 Tầng Đã Triển Khai

Tầng 1: Quant Signal Engine

Lõi định lượng được tách riêng trong Python để bảo đảm AI không trực tiếp phát tín hiệu hoặc tự tính lại chỉ báo kỹ thuật.

  • File chính: bos_quant/signal_engine.py
  • Entry point Docker/service: scripts/quant_service.py
  • CLI batch: scripts/bos_quant_cli.py
  • Test: tests/test_bos_quant.py
  • Document: docs/quant-signal-engine.md

Engine dùng Polars vectorized expressions để tính S_BOS theo Momentum 35%, Volume Delta/CVD 25%, Foreign Flow 25%, Valuation 15%. Output bao gồm score tổng, score từng factor, weighted points, raw factor values, foreign net flow và data quality.

Tầng 2: Financial RAG & LLM Engine

Tầng AI được khóa vai trò: chỉ diễn giải dựa trên output Tầng 1 và dữ liệu RAG có nguồn, không tự bịa tín hiệu mua/bán.

  • Schema: bos_rag/schema.py
  • Prompt hệ thống: bos_rag/prompts.py
  • Payload/engine helper: bos_rag/engine.py
  • Test schema: tests/test_bos_rag_schema.py
  • Document: docs/financial-rag-llm-engine.md

Schema BosRagExplanation dùng extra="forbid", validator bắt buộc s_bos == weight_breakdown.total_s_bos, và mọi catalyst/risk phải có citations.

Tầng 3: Frontend UX/UI & Visualization

Frontend React/TypeScript được đặt trong thư mục frontend/, độc lập với app static gốc để giảm rủi ro khi nâng cấp.

  • Treemap heatmap: frontend/src/components/InteractiveTreemapHeatmap.tsx
  • Ticker chart: frontend/src/components/TickerProfileChart.tsx
  • Virtual table: frontend/src/components/VirtualSignalTable.tsx
  • Type contracts: frontend/src/types/market.ts
  • Document: docs/frontend-visualization-architecture.md

Treemap cho phép đổi size theo vốn hóa/thanh khoản và màu theo giá hoặc S_BOS. Chart dùng TradingView Lightweight Charts với OHLCV và markers mua/bán. Bảng tín hiệu dùng TanStack Virtual để tránh render hàng nghìn DOM rows.

Tầng 4: Omnichannel Alerting

  • Rules engine: api/lib/alert-rules.js
  • Alert endpoint: api/alerts-evaluate.js
  • Service worker: sw.js
  • Frontend Web Push helper: frontend/src/alerts/webPush.ts
  • Document: docs/alerting-api-governance.md

Rules hỗ trợ các điều kiện như S_BOS > 70, RSI < 30, foreign_net_flow crosses_above 0 và crosses_below 0, sẵn sàng nối Web Push, Telegram Bot hoặc Zalo OA webhook.

Tầng 5: API Governance, BFF, Security

  • BFF endpoint: api/bff-dashboard.js
  • Health endpoint: api/health.js
  • Rate limiting: api/lib/rate-limit.js
  • Redis/memory store: api/lib/redis-store.js
  • Circuit breaker: api/lib/circuit-breaker.js

BFF gom Signals + RAG + Alerts trong một request. Token Bucket bảo vệ endpoint nhạy cảm khỏi scraping. Circuit breaker trả stale cache và gắn data_degraded=true khi upstream bị lỗi.

5. Những Vấn Đề Đã Xử Lý Trong Ngày

  • Sửa lỗi JS snapshot trong index.html khiến npm.cmd run check fail vì chuỗi tiếng Việt bị parse sai.
  • Khởi tạo Git repo riêng trong buyorsell-4, gắn remote GitHub và push source lên nhánh main.
  • Gỡ tình trạng domain còn dính record VPS/cũ; chuyển DNS về chuẩn Vercel.
  • Chuyển domain Vercel khỏi project cũ deploy-signal-dashboard sang project mới buyorsell-4.
  • Xử lý lỗi www.buyorsell.vn bị Vercel SSO/protection do domain mapping chưa đúng project; sau khi add domain chính thức, cả www và apex đều 200.
  • Nối GitHub repo vào Vercel project để sẵn sàng auto-deploy từ nhánh main.

6. Rủi Ro Còn Lại Và Khuyến Nghị

Bảo mật domain: Mật khẩu ZoneDNS đã từng được gửi trong chat. Nên đổi lại mật khẩu sau khi cấu hình xong.
  • Nếu muốn dùng Docker/VPS sau này, cần cấu hình lại hạ tầng riêng cho Redis/Quant Service; hiện phương án production chính đang là Vercel để đơn giản SSL và domain.
  • Frontend npm từng báo một moderate audit warning trong dependency tree; chưa chạy npm audit fix --force vì có thể gây breaking changes.
  • Các endpoint realtime/RAG nên được theo dõi thêm bằng synthetic checks định kỳ sau khi traffic thật tăng.
  • Nên cấu hình biến môi trường production trên Vercel cho các API cần secret, không commit secret vào repo.
